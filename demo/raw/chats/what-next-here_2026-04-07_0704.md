---

### 🔜 Natural next steps

1. **Real tokenized data (translation)** — The README already flags this. Swap the sorting dataset for a small translation pair (e.g. Multi30k EN→DE). This means adding:
   - A real tokenizer (SentencePiece or `tiktoken`)
   - A vocabulary builder + a `TranslationDataset` class
   - BLEU evaluation in the eval loop

2. **Learning rate schedule** — You're using a flat Adam LR. The original paper's warmup + inverse-sqrt decay makes a big difference. Easy win:
   ```python
   # Noam-style scheduler
   warmup_steps = 400
   lr_lambda = lambda step: min((step+1)**-0.5, (step+1) * warmup_steps**-1.5)
   ```

3. **Beam search decoding** — `generate()` is currently greedy. Adding beam search (width 3–5) would noticeably improve output quality on real tasks and is a great exercise.

4. **Checkpointing & early stopping** — Save model/optimizer state at best val accuracy, and stop if it plateaus. Essential once training gets longer.

5. **Decoder-only variant (GPT-style)** — You already have `CausalSelfAttention`. A decoder-only `LanguageModel` class that trains on next-token prediction would round out the architecture trio (enc-only, enc-dec, dec-only).

6. **Tests** — There are no unit tests yet. Even a few quick ones (attention output shapes, masking correctness, one-batch overfit) catch regressions fast.

---

### 🧠 Smaller improvements to current code

| What | Why |
|---|---|
| Label smoothing in `CrossEntropyLoss` | Standard trick; 1 line change |
| Mixed precision (`torch.amp`) | Faster training, good practice |
| Configurable weight init | Xavier is fine, but worth experimenting |
| `tqdm` progress bar | Quality-of-life for longer runs |
